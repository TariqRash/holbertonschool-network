What Happens When You Type https://www.google.com and Press Enter

Behind every keystroke lies an intricate chain of networking events, security protocols, and distributed systems working in concert. What feels instant is actually eight distinct stages — each critical, each fascinating.

Step 01

DNS Request

Key Concept

DNS translates human-readable domain names into IP addresses that computers use to identify each other on the network.

Your browser doesn't know where "www.google.com" lives. It only understands IP addresses. So the very first thing it does is perform a DNS lookup to resolve that domain name into a numerical address.

The resolution follows a hierarchy. First, the browser checks its own cache. If it hasn't visited Google recently, it asks the operating system, which checks its local cache. Still no luck? The request goes to your ISP's recursive DNS resolver, which queries the root nameservers, then the .com TLD nameservers, and finally Google's authoritative nameservers. The answer comes back: an A record mapping www.google.com to an IP like 142.250.80.46.

Step 02

TCP/IP

Key Concept

TCP provides reliable, ordered delivery of data. IP handles addressing and routing packets across the internet.

With the IP address in hand, your browser establishes a reliable connection using TCP (Transmission Control Protocol) over IP (Internet Protocol).

TCP uses a three-way handshake: your computer sends a SYN packet to Google's server on port 443 (the standard HTTPS port). The server responds with SYN-ACK, acknowledging your request. Your computer sends an ACK back, confirming the connection. This handshake guarantees both sides are ready for reliable, ordered communication.

Step 03

Firewall

Key Concept

Firewalls filter network traffic based on predefined security rules, blocking unauthorized access while allowing legitimate requests.

Before packets travel freely, they must pass through firewalls on both ends. Your local firewall inspects outgoing traffic to ensure it's allowed. On Google's side, their firewalls examine incoming traffic against massive security rulesets.

Firewalls filter based on IP addresses, port numbers, and protocols. They block suspicious patterns, prevent unauthorized access, and defend against attacks like DDoS. Your HTTPS request to port 443 passes through cleanly — it matches the profile of legitimate web traffic.

Step 04

HTTPS/SSL

Key Concept

TLS encrypts the connection, ensuring data privacy and verifying the server's identity through certificate validation.

Since the URL starts with https://, the browser initiates a TLS handshake before any data is exchanged. Google's server presents its SSL/TLS certificate, signed by a trusted Certificate Authority. Your browser verifies this certificate to confirm it's genuinely talking to Google.

Once trust is established, both sides negotiate a shared symmetric encryption key using asymmetric cryptography. From this point, every byte exchanged is encrypted — protecting your search queries, personal data, and session cookies from anyone listening on the network.

Step 05

Load Balancer

Key Concept

Load balancers distribute incoming requests across multiple servers to ensure no single machine is overwhelmed.

Your request doesn't hit a single server. Google handles billions of requests daily, so when your encrypted packets arrive, they first reach a load balancer.

The load balancer acts as a traffic director, distributing requests across thousands of backend servers using algorithms like Round-Robin or Least Connections. This ensures no server gets overwhelmed, improves response times, and provides fault tolerance — if one server goes down, traffic is automatically redirected to healthy ones.

Step 06

Web Server

Key Concept

The web server handles HTTP requests — serving static files directly or forwarding dynamic requests to the application server.

The load balancer forwards your request to one of Google's web servers (Google uses a custom server called GWS). The web server parses the incoming HTTP request and determines what's being asked for.

For static resources like images or CSS files, the web server serves them directly from disk or cache. For dynamic requests — like loading Google's personalized homepage — it forwards the request to the application server for processing.

Step 07

Application Server

Key Concept

The application server executes business logic — processing queries, applying personalization, and generating dynamic content.

The application server is where business logic lives. It determines your locale, selects the appropriate Google Doodle, prepares personalized settings, and assembles the search interface.

If you type a search query, this is where the search algorithm runs — querying Google's massive index, ranking results, applying personalization, and generating the results page. It orchestrates calls to multiple internal services and databases to build the complete response.

Step 08

Database

Key Concept

Databases store and retrieve persistent data. Google uses Bigtable and Spanner for massive-scale, globally distributed storage.

When the application server needs data — account preferences, search index entries, or cached results — it queries the appropriate database.

Google uses various systems at scale: Bigtable for structured data, Spanner for globally distributed relational data, and caching layers like Memcache for frequently accessed information. The database processes the query, retrieves the data, and returns it to the application server, which assembles everything into an HTML response.

Conclusion

200ms of Engineering

What feels like a single instant — typing a URL and pressing Enter — is actually a beautifully orchestrated sequence of DNS resolution, TCP connections, encryption handshakes, firewall inspections, load balancing, web serving, application processing, and database queries.

All of this happens in roughly 200 to 500 milliseconds. Understanding this flow is fundamental for any software engineer, whether you're debugging latency, designing scalable systems, or simply appreciating the engineering behind the web we use every day.
